
================================================================================
1. ЗАГАЛЬНИЙ ОГЛЯД
================================================================================

WearStore — сучасний інтернет-магазин жіночого одягу з елегантним дизайном, повною українською локалізацією та зручним інтерфейсом для покупців і адміністрування. Платформа орієнтована на українських споживачів та забезпечує повний цикл електронної комерції від перегляду каталогу до оплати та доставки.


================================================================================
2. ТЕХНОЛОГІЧНИЙ СТЕК
================================================================================

FRONTEND

Основою клієнтської частини є React версії 18 з використанням TypeScript для забезпечення типізації. Збірка та локальний сервер розробки реалізовані на Vite 5. Для управління серверним станом та кешування використовується TanStack Query версії 5, а для клієнтського стану (кошик, UI-елементи) — Zustand 4. Маршрутизація побудована на React Router 6.

Стилізація виконується за допомогою Tailwind CSS 3, анімації реалізовані через Framer Motion 11. Робота з формами здійснюється через React Hook Form 7 у поєднанні з Zod 3 для валідації. HTTP-запити виконуються через Axios.


BACKEND

Серверна частина працює на Node.js 20 LTS з фреймворком Express 4. Код написаний на TypeScript. Взаємодія з базою даних реалізована через ORM Prisma 5. Основна база даних — PostgreSQL 16, для кешування та зберігання сесій використовується Redis 7.

Автентифікація побудована на JWT з механізмом Refresh Tokens. Паролі хешуються алгоритмом Argon2. Завантаження та обробка зображень виконуються через Multer та Sharp. Email-сповіщення відправляються через Nodemailer. Валідація вхідних запитів здійснюється через Zod. Безпека забезпечується Helmet та налаштуванням CORS. Логування реалізоване через Winston.


ІНФРАСТРУКТУРА

Контейнеризація виконана через Docker та Docker Compose. Nginx виступає reverse proxy та обслуговує статичні файли. Зображення зберігаються у хмарному сховищі AWS S3 або Cloudflare R2. Платіжна система — LiqPay API.


================================================================================
3. АРХІТЕКТУРА СИСТЕМИ
================================================================================

Архітектура побудована за принципом розділення клієнтської та серверної частин. Всі запити від браузерів та мобільних пристроїв надходять через HTTPS на Nginx, який виконує SSL termination, gzip-стиснення та роздачу статичних файлів.

Nginx розподіляє трафік між React SPA (статична збірка Vite) та Express API Server. React-додаток відповідає за відображення каталогу, кошика, особистого кабінету та адмін-панелі.

Express API Server має багатошарову архітектуру. Middleware Layer обробляє автентифікацію, валідацію та логування. Controllers приймають HTTP-запити та делегують бізнес-логіку відповідним сервісам. Services Layer містить всю бізнес-логіку додатку. Prisma ORM забезпечує типобезпечну взаємодію з базою даних.

Дані зберігаються у трьох системах: PostgreSQL для основних даних, Redis для кешу та сесій, S3/R2 для зображень товарів.


================================================================================
4. МОДЕЛЬ ДАНИХ
================================================================================

КОРИСТУВАЧІ ТА АВТЕНТИФІКАЦІЯ

Модель User зберігає основну інформацію про користувачів: унікальний ідентифікатор (cuid), email, хеш пароля, ім'я та прізвище, телефон, роль (CLIENT, MANAGER, ADMIN), статус (ACTIVE, BANNED), URL аватара та часові мітки створення й оновлення. Користувач може мати кілька адрес доставки, замовлень, один кошик, список бажань та кілька refresh tokens.

Модель RefreshToken зберігає токени оновлення з прив'язкою до користувача, терміном дії та часом створення. При видаленні користувача всі його токени видаляються каскадно.

Модель Address зберігає адреси доставки: мітка (наприклад, "Дім" чи "Робота"), місто, вулиця, будинок, квартира, поштовий індекс та прапорець адреси за замовчуванням.


КАТАЛОГ ТОВАРІВ

Модель Category реалізує ієрархічну структуру категорій. Кожна категорія має назву, унікальний slug, опис, URL зображення, посилання на батьківську категорію та дочірні категорії, порядок сортування та прапорець активності.

Модель Product є центральною сутністю каталогу. Товар має SKU, назву, slug, повний та короткий опис, прив'язку до категорії, базову ціну та ціну зі знижкою. Додаткові поля включають матеріал та інструкції з догляду. Прапорці isFeatured та isActive визначають відображення товару. Товар має колекції зображень, варіантів та записів списку бажань.

Модель ProductImage зберігає зображення товарів з URL, альтернативним текстом, порядком сортування та прапорцем головного зображення.

Модель ProductVariant описує варіанти товару за розміром (XS, S, M, L, XL, XXL) та кольором. Кожен варіант має залишок на складі та опціональну надбавку до базової ціни. Унікальність забезпечується комбінацією товару, розміру та кольору.

Модель WishlistItem створює зв'язок між користувачем та товаром у списку бажань з унікальністю пари.


КОШИК

Модель Cart прив'язана до користувача один-до-одного та містить колекцію товарів.

Модель CartItem зберігає позицію кошика з посиланням на варіант товару та кількістю. Унікальність забезпечується парою кошик-варіант.


ЗАМОВЛЕННЯ

Модель Order зберігає повну інформацію про замовлення. Номер замовлення формується за шаблоном WS-2024-000001. Статус замовлення може бути: PENDING (очікує підтвердження), CONFIRMED (підтверджено), PROCESSING (обробляється), SHIPPED (відправлено), DELIVERED (доставлено), CANCELLED (скасовано), REFUNDED (повернуто кошти).

Фінансова інформація включає підсумок, суму знижки, вартість доставки та загальну суму. Поле promoCode зберігає застосований промокод.

Доставка описується методом (NOVA_POSHTA_WAREHOUSE, NOVA_POSHTA_COURIER, UKRPOSHTA), посиланням на адресу, знімком адреси у JSON та трек-номером.

Оплата характеризується методом (LIQPAY, CASH_ON_DELIVERY), статусом (PENDING, PAID, FAILED, REFUNDED), ідентифікатором транзакції LiqPay та часом оплати.

Примітки клієнта та менеджера зберігаються окремо. Історія зміни статусів відстежується у пов'язаній моделі.

Модель OrderItem зберігає позицію замовлення зі знімком даних товару на момент оформлення, кількістю, ціною за одиницю та загальною ціною позиції.

Модель OrderStatusHistory фіксує кожну зміну статусу замовлення з попереднім та новим статусом, коментарем та ідентифікатором менеджера.


ПРОМОКОДИ

Модель PromoCode визначає знижки за кодом. Тип може бути PERCENTAGE (відсотковий) або FIXED_AMOUNT (фіксована сума). Поля включають значення знижки, мінімальну суму замовлення, максимальну кількість використань, поточну кількість використань, період дії та прапорець активності.

5. API ENDPOINTS
================================================================================

АВТЕНТИФІКАЦІЯ (/api/auth)

POST /register — реєстрація нового користувача, публічний доступ.
POST /login — вхід в систему, публічний доступ.
POST /logout — вихід з інвалідацією токенів, потребує авторизації.
POST /refresh — оновлення access token за refresh token, публічний доступ.
POST /forgot-password — запит на скидання пароля, публічний доступ.
POST /reset-password — встановлення нового пароля за токеном, публічний доступ.
GET /me — отримання даних поточного користувача, потребує авторизації.


КАТАЛОГ (/api/catalog)

GET /categories — отримання дерева категорій, публічний доступ.
GET /categories/:slug — отримання категорії за slug, публічний доступ.
GET /products — список товарів з фільтрацією та пагінацією, публічний доступ.
GET /products/:slug — деталі товару за slug, публічний доступ.
GET /products/featured — рекомендовані товари, публічний доступ.
GET /search — пошук товарів, публічний доступ.

Параметри фільтрації для /products включають: category (slug категорії), minPrice та maxPrice (діапазон цін), sizes (масив розмірів), colors (масив кольорів), inStock (тільки в наявності), onSale (тільки зі знижкою), sort (price_asc, price_desc, newest, popular), page та limit (пагінація).


КОШИК (/api/cart)

GET / — отримання кошика поточного користувача.
POST /items — додавання товару до кошика.
PATCH /items/:id — зміна кількості товару.
DELETE /items/:id — видалення товару з кошика.
DELETE / — очищення кошика.
POST /apply-promo — застосування промокоду.

Всі ендпоінти кошика потребують авторизації.


ЗАМОВЛЕННЯ (/api/orders)

POST / — створення замовлення з кошика, доступ для клієнта.
GET / — список замовлень поточного користувача, доступ для клієнта.
GET /:id — деталі замовлення, доступ для власника замовлення.
POST /:id/cancel — скасування замовлення, доступ для власника.


ОПЛАТА (/api/payments)

POST /liqpay/create — ініціювання оплати через LiqPay, потребує авторизації.
POST /liqpay/callback — webhook від LiqPay, верифікація за підписом.
GET /liqpay/status/:orderId — перевірка статусу оплати, потребує авторизації.


СПИСОК БАЖАНЬ (/api/wishlist)

GET / — отримання списку бажань.
POST /:productId — додавання товару до списку.
DELETE /:productId — видалення товару зі списку.

Всі ендпоінти потребують авторизації.


ПРОФІЛЬ (/api/profile)

GET / — дані профілю користувача.
PATCH / — оновлення профілю.
PATCH /password — зміна пароля.
GET /addresses — список адрес доставки.
POST /addresses — додавання нової адреси.
PATCH /addresses/:id — редагування адреси.
DELETE /addresses/:id — видалення адреси.

Всі ендпоінти потребують авторизації.

================================================================================
7. СТРУКТУРА FRONTEND
================================================================================

СТОРІНКИ ТА МАРШРУТИ

Головна сторінка доступна за адресою /. Каталог товарів розташований за адресою /catalog, а товари конкретної категорії — /catalog/:categorySlug. Картка окремого товару доступна за /product/:productSlug. Кошик знаходиться за адресою /cart, оформлення замовлення — /checkout, сторінка успішного замовлення — /checkout/success. Результати пошуку відображаються на /search.

Автентифікація включає сторінки: /auth/login для входу, /auth/register для реєстрації, /auth/forgot-password для запиту відновлення пароля, /auth/reset-password/:token для скидання пароля за токеном.

Особистий кабінет розташований за адресою /account. Мої замовлення доступні на /account/orders, деталі конкретного замовлення — /account/orders/:id. Список бажань знаходиться на /account/wishlist, адреси доставки — /account/addresses, налаштування профілю — /account/settings.

Адмін-панель має головну сторінку з дашбордом за адресою /admin. Управління товарами здійснюється через /admin/products, створення нового товару — /admin/products/new, редагування — /admin/products/:id/edit. Категорії доступні на /admin/categories. Замовлення відображаються на /admin/orders, деталі замовлення — /admin/orders/:id. Управління користувачами (тільки для адміна) знаходиться на /admin/users. Промокоди доступні на /admin/promo-codes, звіти (тільки для адміна) — /admin/reports.


СТРУКТУРА КОМПОНЕНТІВ

Кореневий каталог src містить директорію app з файлами App.tsx (головний компонент), router.tsx (налаштування маршрутизації) та providers.tsx (провайдери контексту).

Директорія pages містить піддиректорії для кожної групи сторінок: home, catalog, product, cart, checkout, auth, account та admin.

Директорія features організована за функціональними модулями: auth (автентифікація), catalog (каталог), cart (кошик), orders (замовлення), wishlist (список бажань) та admin (адміністрування). Кожен модуль містить власні піддиректорії api, components, hooks та store.

Директорія shared містить спільний код: api (HTTP-клієнт та типи), components (ui для базових компонентів, layout для Header, Footer, Sidebar, common для ProductCard, Pagination тощо), hooks, lib, types та utils.

Директорія assets зберігає fonts, icons та images. Директорія styles містить globals.css з глобальними стилями.


================================================================================
8. ДИЗАЙН-СИСТЕМА
================================================================================

КОЛЬОРОВА ПАЛІТРА

Основна кольорова гама побудована на рожево-пурпурних відтінках для створення жіночного та елегантного вигляду. Найсвітліший відтінок primary-50 має код #fdf2f4, найтемніший primary-900 — #7a1f3b. Основний акцентний колір primary-500 має код #e05070.

Нейтральні кольори від neutral-50 (#fafafa) до neutral-900 (#171717) використовуються для тексту, фонів та розділювачів.

Додаткові акцентні кольори включають золотий (#c9a962) для преміум-елементів, зелений (#22c55e) для успішних операцій, жовтогарячий (#f59e0b) для попереджень та червоний (#ef4444) для помилок.


ТИПОГРАФІКА

Для заголовків використовується шрифт Playfair Display із засічками, який надає елегантності та вишуканості. Основний текст набирається шрифтом Inter без засічок для оптимальної читабельності на екранах.

Розміри тексту варіюються від xs (0.75rem / 12px) для дрібних підписів до 5xl (3rem / 48px) для великих заголовків. Базовий розмір тексту — 1rem (16px).


КОМПОНЕНТИ UI

Кнопки реалізовані у кількох варіантах. Primary має рожевий фон з білим текстом для основних дій. Secondary має прозорий фон з рожевою рамкою для другорядних дій. Ghost — без рамки з hover-ефектом для допоміжних дій. Всі кнопки доступні у трьох розмірах: sm, md та lg.

Картки товарів мають плавну тінь при наведенні, кнопку швидкого перегляду (Quick View), іконку серця для додавання до списку бажань та бейджі "Новинка" або "Знижка -X%".

Форми оформлені у мінімалістичному стилі з плаваючими labels, які піднімаються при фокусі, та іконками валідації для зворотного зв'язку.


================================================================================
9. БЕЗПЕКА
================================================================================

АВТЕНТИФІКАЦІЯ

Access Token реалізований як JWT з терміном дії 15 хвилин. Токен зберігається в пам'яті JavaScript для захисту від XSS-атак.

Refresh Token має термін дії 7 днів та зберігається в HttpOnly cookie, недоступній для JavaScript. При кожному оновленні токенів виконується ротація refresh token для захисту від компрометації.

Система обмежує кількість активних сесій до 5 пристроїв одночасно. При перевищенні ліміту найстаріша сесія автоматично завершується.


АВТОРИЗАЦІЯ

Middleware перевірки ролей requireRole приймає масив дозволених ролей та порівнює з роллю поточного користувача. При невідповідності повертається помилка 403 Forbidden з повідомленням "Недостатньо прав доступу".


ЗАХИСТ API

Rate limiting обмежує кількість запитів: 100 на хвилину для автентифікованих користувачів та 30 для гостей.

CORS налаштований з whitelist дозволених доменів. Helmet встановлює заголовки безпеки.

Валідація всіх вхідних даних виконується через Zod перед обробкою. Prisma використовує параметризовані запити для захисту від SQL-ін'єкцій. На фронтенді застосовується DOMPurify для захисту від XSS.


ЗАХИСТ ПЛАТЕЖІВ

Всі callback-запити від LiqPay верифікуються за підписом для підтвердження автентичності. Обробка платежів реалізована з ідемпотентністю для запобігання подвійному списанню. Всі транзакції логуються для аудиту.


================================================================================
10. ПРОДУКТИВНІСТЬ
================================================================================

КЕШУВАННЯ

Redis використовується для кешування з різними TTL залежно від типу даних. Категорії кешуються на 1 годину (3600 секунд), оскільки змінюються рідко. Списки товарів кешуються на 5 хвилин (300 секунд). Деталі окремих товарів — на 10 хвилин (600 секунд). Сесії користувачів зберігаються 24 години (86400 секунд).


ОПТИМІЗАЦІЯ ЗОБРАЖЕНЬ

При завантаженні зображення автоматично генеруються responsive-варіанти для різних розмірів екранів: 320w для мобільних, 640w для планшетів, 1024w для десктопів та 1920w для великих моніторів.

Основний формат — WebP з fallback на JPEG для застарілих браузерів. Реалізовано lazy loading з blur placeholder для покращення сприйняття швидкості завантаження. Доставка зображень здійснюється через CDN Cloudflare.


FRONTEND ОПТИМІЗАЦІЇ

Code splitting по маршрутах забезпечує завантаження тільки необхідного коду для поточної сторінки. Prefetch критичних ресурсів прискорює навігацію. Optimistic updates для кошика забезпечують миттєвий відгук інтерфейсу. Virtual scrolling застосовується для довгих списків товарів.


================================================================================
11. EMAIL-СПОВІЩЕННЯ
================================================================================

ШАБЛОНИ ЛИСТІВ

При реєстрації клієнт отримує лист "Ласкаво просимо до WearStore!" з привітанням та корисною інформацією.

Цикл замовлення супроводжується серією листів клієнту: "Замовлення #WS-2024-000001 прийнято" при створенні, "Замовлення #WS-2024-000001 підтверджено" при підтвердженні менеджером, "Ваше замовлення в дорозі!" при відправленні з трек-номером, "Замовлення доставлено" при отриманні.

Для скидання пароля надсилається лист "Відновлення пароля" з посиланням на сторінку скидання.

Менеджери отримують сповіщення "Нове замовлення #WS-2024-000001" при надходженні нових замовлень.


================================================================================
12. СТРУКТУРА ПРОЄКТУ
================================================================================

Проєкт організований як monorepo з використанням pnpm workspaces.

Кореневий каталог wearstore містить директорію apps з двома додатками. Директорія web містить React frontend з власними src, public, index.html, vite.config.ts, tailwind.config.js та package.json. Директорія api містить Express backend з src (config, controllers, services, middleware, routes, utils, types, index.ts), prisma (schema.prisma, migrations, seed.ts) та package.json.

Директорія packages містить shared — спільний пакет з типами, валідаторами та константами, що використовуються обома додатками.

Директорія docker містить Dockerfile.web для збірки фронтенду, Dockerfile.api для збірки бекенду та nginx.conf для налаштування reverse proxy.

Кореневі файли включають docker-compose.yml для розробки, docker-compose.prod.yml для продакшену, .env.example з прикладами змінних оточення, package.json, pnpm-workspace.yaml та README.md.


================================================================================
13. ЗМІННІ ОТОЧЕННЯ
================================================================================

DATABASE_URL — рядок підключення до PostgreSQL у форматі postgresql://user:pass@localhost:5432/wearstore.

REDIS_URL — рядок підключення до Redis у форматі redis://localhost:6379.

JWT_ACCESS_SECRET та JWT_REFRESH_SECRET — секретні ключі для підпису JWT токенів.

LIQPAY_PUBLIC_KEY та LIQPAY_PRIVATE_KEY — ключі API LiqPay для інтеграції платежів.

S3_ENDPOINT — URL ендпоінта S3-сумісного сховища. S3_ACCESS_KEY та S3_SECRET_KEY — ключі доступу. S3_BUCKET — назва бакета для зображень. S3_PUBLIC_URL — публічний URL для доступу до зображень.

SMTP_HOST, SMTP_PORT, SMTP_USER та SMTP_PASS — налаштування SMTP-сервера для відправки email.

NODE_ENV — оточення (development, production). API_PORT — порт API-сервера. FRONTEND_URL — URL фронтенду. API_URL — URL API-сервера.

14. ВИСНОВОК
================================================================================

Дана специфікація описує повнофункціональний інтернет-магазин жіночого одягу WearStore з сучасним технологічним стеком PERN (PostgreSQL, Express, React, Node.js), доповненим TypeScript, Prisma, Redis та іншими сучасними інструментами.

Архітектура забезпечує чітке розділення відповідальностей, масштабованість, безпеку та високу продуктивність. Рольова модель з трьома рівнями доступу (клієнт, менеджер, адміністратор) покриває всі потреби електронної комерції.

Дизайн-система з елегантною кольоровою палітрою та продуманою типографікою створює сучасний жіночний вигляд магазину. Повна українська локалізація та інтеграція з українською платіжною системою LiqPay забезпечують зручність для українських користувачів.
